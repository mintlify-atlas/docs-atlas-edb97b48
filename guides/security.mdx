title: "Security hardening the room middleware for production"
description: "How room defends against cookie theft, queue flooding, poll abuse, and ghost tickets — plus the settings you must configure before going to production."
---

`room` is designed with a threat model that assumes adversarial clients: clients that replay cookies, flood the queue, poll faster than allowed, abandon tickets intentionally, and manipulate pass cookies. The library handles most of these automatically. A small number of settings — primarily `SetSecureCookie`, `SetMaxQueueDepth`, and cookie scoping — require explicit configuration for production deployments.

## Security properties

| Concern | How `room` handles it |
|---|---|
| **Cookie theft / replay** | Tokens are 128-bit cryptographically random hex strings (`crypto/rand`, 16 bytes). `HttpOnly` flag prevents XSS reads. Call `SetSecureCookie(true)` for the `Secure` flag on HTTPS deployments. |
| **Queue flooding** | `SetMaxQueueDepth(n)` rejects new arrivals with `503 Service Unavailable` when the queue exceeds `n`, preventing unbounded memory growth. Default is 0 (unlimited). |
| **Poll abuse** | Per-token rate limiting on `/queue/status` — polls arriving faster than 1 request/second per token receive `429 Too Many Requests` with a `Retry-After: 1` header. |
| **Ghost tickets** | The reaper runs on a configurable interval (default 5 minutes), evicts expired tokens, and advances `nowServing` so the queue does not stall behind abandoned clients. |
| **Cookie scoping** | `SetCookiePath` and `SetCookieDomain` restrict cookie visibility in multi-app deployments. `SameSite=Lax` is set by default on all session cookies. |
| **Capacity enforcement** | The `nowServing` window guard prevents the serving window from inflating beyond `cap` even under adversarial client disconnection patterns. |
| **Promotion serialization** | `PromoteToken` acquires a dedicated mutex (`promoteMu`) and uses a monotonic insert counter (`promoteInsert`) to guarantee unique ticket assignment under concurrent promotions. |
| **Pass expiry** | VIP passes have a server-side expiry check — not just cookie `MaxAge`. The reaper sweeps expired passes on every cycle. Expired `room_pass` cookies are silently ignored. |

## Token entropy

Every `room_ticket` and `room_pass` token is generated by `generateToken`, which reads 16 bytes from `crypto/rand` and hex-encodes them:

```go
func generateToken() (string, error) {
    b := make([]byte, tokenBytes) // tokenBytes = 16
    if _, err := rand.Read(b); err != nil {
        return "", err
    }
    return hex.EncodeToString(b), nil
}
```

16 bytes = 128 bits of entropy. Brute-forcing a valid token at 1 million guesses per second would take approximately 10²⁵ years. Guessing a valid token is not a practical attack.

## Production checklist

<Steps>

### Enable the Secure cookie flag for HTTPS deployments

The `Secure` flag tells browsers to send cookies only over HTTPS, preventing token leakage over plain HTTP. The default is `false` so that local development works without configuration.

```go
wr.SetSecureCookie(true)
```

Set this for any deployment served over HTTPS, including deployments behind TLS-terminating proxies (Cloudflare, nginx, AWS ALB) where `c.Request.TLS` may be `nil` even though the end-user connection is encrypted.

### Set a max queue depth to cap memory usage

An unlimited queue is an unbounded memory allocation. Set a depth limit appropriate for your server's available memory. Clients beyond the limit receive `503 Service Unavailable` immediately.

```go
// Reject new arrivals when more than 10,000 clients are waiting.
if err := wr.SetMaxQueueDepth(10000); err != nil {
    log.Fatal(err)
}
```

Estimate your budget: each queued token consumes a small fixed amount of memory (a map entry with a 32-byte hex string key and a `ticketEntry` struct). 100,000 queued tokens typically consume less than 50 MB, but under sustained flood conditions without a limit you have no bound.

### Scope cookies for multi-app deployments

If multiple apps share a domain, restrict cookie scope so a token issued by one app is not sent to another:

```go
// Scope to a specific path prefix.
wr.SetCookiePath("/shop")

// Scope to a specific subdomain (or set of subdomains).
wr.SetCookieDomain("shop.example.com")
```

Leaving both at their defaults (`"/"` and `""`) is correct for single-app deployments. For multi-app deployments on the same domain, scoping prevents one app's cookies from interfering with another's queue.

### Register payment routes before RegisterRoutes

Payment handlers must bypass the waiting room — if they don't, a client trying to pay is stuck in the queue they are trying to skip, creating a deadlock.

```go
// Payment handlers are registered BEFORE RegisterRoutes.
// They are outside the queue — no room_ticket cookie is required.
r.GET("/queue/purchase", handlePurchasePage)
r.POST("/queue/purchase/confirm", handlePurchaseConfirm)
r.POST("/stripe/webhook", handleStripeWebhook)

// All routes registered after this line are gated by the waiting room.
wr.RegisterRoutes(r)
```

The same rule applies to any health check, admin endpoint, or monitoring scrape target that should not be queued.

</Steps>

## Reaper configuration

The reaper is your defence against ghost tickets — clients who entered the queue and then closed their browser without being admitted. Ghost tickets block forward progress: `nowServing` cannot advance past a ticket that is never consumed, so every client behind a ghost is stuck indefinitely.

The reaper evicts tokens older than 30 minutes (the cookie TTL) and advances `nowServing` by the number of evicted ghost tickets. Active pollers have their token's `issuedAt` refreshed on every `/queue/status` call, so they are never evicted under normal operation.

```go
// Default: 5 minutes. Valid range: 5 seconds – 24 hours.
// Tighter intervals reduce ghost-ticket stall windows at the cost of
// more frequent lock contention on the token store.
if err := wr.SetReaperInterval(30 * time.Second); err != nil {
    log.Fatal(err)
}
```

Tighten the interval for high-traffic events where ghost tickets accumulate quickly. Loosen it for low-traffic background queues where memory pressure is not a concern.

## Poll rate limiting

`/queue/status` is rate-limited per token to 1 request per second. Clients polling faster receive:

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 1
```

This is enforced in `StatusHandler` using the token's `lastPoll` timestamp, updated atomically on each valid poll. No external rate limiter or middleware is required.

## CORS for cross-origin deployments

If your waiting room page is served from a different origin than your API, browsers will send preflight `OPTIONS` requests to `/queue/status` before the `fetch()` polling call. `RegisterRoutes` automatically registers an `OPTIONS` handler:

```go
// RegisterRoutes does this for you:
r.OPTIONS("/queue/status", func(c *gin.Context) {
    c.Status(http.StatusNoContent)
})
r.GET("/queue/status", wr.StatusHandler())
r.Use(wr.Middleware())
```

If you are registering routes manually instead of using `RegisterRoutes`, add the `OPTIONS` handler explicitly, or configure your CORS middleware to handle preflight for `/queue/status`.

<Warning>
Do not put `/queue/status` behind authentication middleware. The polling fetch call from the waiting room page runs before the user is admitted, so it has no session credentials. Authentication middleware that rejects unauthenticated requests will break the polling loop and prevent clients from ever being admitted.
</Warning>

## Pass security

VIP pass tokens are subject to the same 128-bit entropy guarantee as queue tokens — they are generated by the same `generateToken` function. The pass expiry is checked server-side on every request via `HasValidPass`:

```go
// HasValidPass checks the server-side passStore, not just the cookie.
// An expired or replayed room_pass cookie is silently ignored.
if wr.HasValidPass(passCookie.Value) {
    // auto-promote
}
```

The reaper sweeps the `passStore` on every reap cycle. Expired passes are removed from the server-side store, so even a non-expired browser cookie is rejected after the server-side record is cleaned up.
