---
title: "Get started with room: add a waiting room to Gin apps"
description: "Install room, initialize a WaitingRoom, register routes on your Gin engine, and verify that excess traffic is queued — all in a single Go file."
---

This guide walks through adding a FIFO waiting room to an existing Gin application from scratch. By the end you will have a running server where the 501st concurrent request sees a live queue page instead of an error, and is admitted automatically when a slot opens.

<Steps>
  <Step title="Install the package">
    Add `room` to your Go module. It has one transitive dependency — [sema](https://github.com/andreimerlescu/sema) — and requires Go 1.22 or later.

    ```bash
    go get github.com/andreimerlescu/room
    ```

    Verify the import path resolves:

    ```bash
    go list -m github.com/andreimerlescu/room
    ```
  </Step>

  <Step title="Initialize and register routes">
    The minimal setup is three method calls. Create a `WaitingRoom`, call `Init` with your capacity, and pass your `*gin.Engine` to `RegisterRoutes`. Every route registered **after** `RegisterRoutes` is gated by the waiting room.

    ```go main.go
    package main

    import (
        "log"

        "github.com/andreimerlescu/room"
        "github.com/gin-gonic/gin"
    )

    func main() {
        r := gin.Default()

        wr := &room.WaitingRoom{}
        if err := wr.Init(500); err != nil {
            log.Fatal(err)
        }
        defer wr.Stop()

        // Registers GET /queue/status and attaches the middleware.
        // Routes added after this call are gated.
        wr.RegisterRoutes(r)

        r.GET("/", func(c *gin.Context) {
            c.String(200, "You're in!")
        })

        r.Run(":8080")
    }
    ```

    `RegisterRoutes` does three things in the correct order:

    1. Registers `OPTIONS /queue/status` for CORS preflight support.
    2. Registers `GET /queue/status` — the polling endpoint used by the waiting room page.
    3. Attaches `wr.Middleware()` via `r.Use(...)` so all subsequent routes are gated.

    <Note>`defer wr.Stop()` cancels the background reaper goroutine on shutdown. Always include it to avoid leaked goroutines, especially in tests or embedded servers.</Note>
  </Step>

  <Step title="Verify the waiting room activates">
    Start the server, then open `http://localhost:8080/` in a browser or send a request with `curl`:

    ```bash
    go run main.go
    ```

    ```bash
    curl -i http://localhost:8080/
    ```

    When capacity is exhausted the server responds with `200 OK` and an HTML page showing the client's queue position. The page polls `GET /queue/status` every 3 seconds and reloads automatically when `ready=true` is returned.

    The `/queue/status` JSON payload looks like this while a client is waiting:

    ```json
    {
      "ready": false,
      "position": 4,
      "utilization": 1.0
    }
    ```

    And like this when the client is ready to be admitted:

    ```json
    {
      "ready": true
    }
    ```

    <Tip>Open `http://localhost:8080/` in a browser while running the [sample load test](https://github.com/andreimerlescu/room/tree/main/sample/basic-web-app) (`bash test.sh`) to see the position counter tick down in real time.</Tip>
  </Step>

  <Step title="Production hardening and next steps">
    The default configuration is designed for local development. Before deploying to production, apply these two settings:

    ```go
    // Set the Secure flag on session cookies. Required for HTTPS deployments,
    // including those behind a TLS-terminating proxy (nginx, Cloudflare, ALB).
    wr.SetSecureCookie(true)

    // Prevent unbounded memory growth — reject arrivals beyond this queue depth.
    // New arrivals receive 503 Service Unavailable when the limit is reached.
    if err := wr.SetMaxQueueDepth(10000); err != nil {
        log.Fatal(err)
    }
    ```

    For simple cases where you do not need to retain a reference to `WaitingRoom`, use `NewWaitingRoom`. It registers `/queue/status` and returns the middleware handler in a single call. Note that `NewWaitingRoom` starts a background goroutine tied to the process lifetime — there is no way to call `Stop()`:

    ```go
    r.Use(room.NewWaitingRoom(r, 500))
    ```
